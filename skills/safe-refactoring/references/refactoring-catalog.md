安全性が高く、頻繁に使用されるものを中心に記載。

## Chapter 6: 基本リファクタリング

### Extract Function（関数の抽出）

- **目的**: コードの断片を理解し、その目的に基づいて名前を付けた関数に抽出
- **適用シナリオ**: コードの意図を理解するのに時間がかかる場合。6行以上は検討対象
- **手順**:
  1. 新しい関数を作成し、目的に基づいて名前を付ける
  2. 抽出したいコードを新しい関数にコピー
  3. 参照される変数をパラメータまたはローカル変数として調整
  4. 元のコードを関数呼び出しに置き換える
  5. テストを実行

### Inline Function（関数のインライン化）

- **目的**: 関数本体が名前と同じくらい明確な場合、関数を削除してインライン化
- **適用シナリオ**: 過度の間接化により読みづらくなっている場合
- **手順**:
  1. 関数がポリモーフィックでないことを確認
  2. すべての呼び出し元を見つける
  3. 各呼び出しを関数本体に置き換える
  4. 元の関数を削除

### Extract Variable（変数の抽出）

- **目的**: 式に名前を付けて理解しやすくする
- **適用シナリオ**: 複雑な式を分解して理解しやすくする場合
- **手順**:
  1. 抽出する式に副作用がないことを確認
  2. 変更不可の変数を宣言し、式のコピーを代入
  3. 元の式を新しい変数に置き換える
  4. テストを実行

### Inline Variable（変数のインライン化）

- **目的**: 変数名が式自体よりも説明的でない場合に削除
- **適用シナリオ**: 変数がリファクタリングを妨げている場合
- **手順**:
  1. 代入の右辺に副作用がないことを確認
  2. 変数への参照をすべて式に置き換える
  3. テストを実行

### Change Function Declaration（関数宣言の変更）

- **目的**: 関数名の変更、パラメータの追加・削除
- **適用シナリオ**: より良い名前を思いついた場合、パラメータリストを整理する場合
- **手順**:
  1. パラメータを削除する場合、参照がないことを確認
  2. 関数本体を変更
  3. すべての呼び出し元を更新
  4. テストを実行

### Encapsulate Variable（変数のカプセル化）

- **目的**: データへの直接アクセスをgetter/setter関数経由に変更
- **適用シナリオ**: データの参照と更新を制御したい場合
- **手順**:
  1. カプセル化する変数への参照・更新関数を作成
  2. すべての参照を関数呼び出しに変更
  3. 変数の可視性を制限
  4. テストを実行

### Rename Variable（変数名の変更）

- **目的**: より適切な名前に変更して理解しやすくする
- **適用シナリオ**: 変数の役割がより明確になる名前がある場合
- **手順**:
  1. 広くアクセスされる変数の場合、Encapsulate Variableを検討
  2. すべての参照を新しい名前に変更
  3. テストを実行

### Introduce Parameter Object（パラメータオブジェクトの導入）

- **目的**: 繰り返し出現するパラメータグループをオブジェクトに置き換える
- **適用シナリオ**: 同じパラメータセットが複数のメソッドに出現する場合
- **手順**:
  1. パラメータを保持する新しいクラスを作成
  2. メソッドにパラメータオブジェクトを受け取るパラメータを追加
  3. 古いパラメータを削除し、パラメータオブジェクトのフィールドに置き換え
  4. 関連ロジックをパラメータオブジェクトに移動することを検討

## Chapter 7: カプセル化

### Encapsulate Record（レコードのカプセル化）

- **目的**: データ構造を隠蔽し、アクセスを制御
- **適用シナリオ**: 可変データ構造を直接公開している場合
- **手順**:
  1. レコードを保持するクラスを作成
  2. レコードを返すメソッドを追加
  3. すべての参照を新しいメソッド経由に変更
  4. レコードへのアクセスをカプセル化

### Encapsulate Collection（コレクションのカプセル化）

- **目的**: コレクションへの直接アクセスを防ぎ、コピーを返す
- **適用シナリオ**: コレクションを直接返すgetterがある場合
- **手順**:
  1. コレクションのコピーを返すメソッドを作成
  2. 要素の追加・削除メソッドを提供
  3. 直接アクセスを新しいメソッドに置き換え

### Replace Primitive with Object（プリミティブからオブジェクトへ）

- **目的**: プリミティブ型を意味のあるドメインオブジェクトに置き換える
- **適用シナリオ**: 電話番号、金額、座標などをプリミティブで扱っている場合
- **手順**:
  1. Encapsulate Variableを適用
  2. 新しい適切な名前のクラスを作成
  3. フィールドを新しいクラスのインスタンスに変更
  4. 関連するロジックを新しいクラスに移動

### Replace Temp with Query（一時変数をクエリに置き換え）

- **目的**: 一時変数を計算を行うメソッドに置き換える
- **適用シナリオ**: 一時変数が多くてExtract Functionが困難な場合
- **手順**:
  1. 一時変数が1回だけ代入され、副作用がないことを確認
  2. 式を返すメソッドを抽出
  3. 一時変数への参照をメソッド呼び出しに置き換え

### Extract Class（クラスの抽出）

- **目的**: 1つのクラスが2つの責任を持っている場合に分割
- **適用シナリオ**: クラスが大きすぎる、または複数の責任を持っている場合
- **手順**:
  1. 責任の分割方法を決定
  2. 新しいクラスを作成
  3. 元のクラスとの関係を作成
  4. Move FieldとMove Methodで機能を移動

### Inline Class（クラスのインライン化）

- **目的**: ほとんど何もしないクラスを削除
- **適用シナリオ**: Extract Classをやりすぎた場合、クラスの責任が減った場合
- **手順**:
  1. インライン先のクラスに機能を移動
  2. すべての参照を更新
  3. 元のクラスを削除

### Hide Delegate（委譲の隠蔽）

- **目的**: デリゲートオブジェクトへのアクセスを隠蔽
- **適用シナリオ**: `person.department.manager` のような連鎖呼び出しがある場合
- **手順**:
  1. デリゲートの各メソッドに対応する委譲メソッドを作成
  2. クライアントを新しいメソッド使用に変更
  3. デリゲートへのアクセサを削除

### Remove Middle Man（中間者の削除）

- **目的**: 過度な委譲メソッドを削除
- **適用シナリオ**: クラスが単なる委譲しかしていない場合
- **手順**:
  1. デリゲートオブジェクトへのアクセサを作成
  2. 委譲メソッドを削除し、クライアントを直接呼び出しに変更

## Chapter 8: 機能の移動

### Move Function（関数の移動）

- **目的**: 関数をより適切なコンテキストに移動
- **適用シナリオ**: 関数が別のクラスのデータをより多く使用している場合
- **手順**:
  1. 移動先のコンテキストで新しい関数を作成
  2. 元の関数から新しい関数にコードをコピー
  3. 元の関数を新しい関数への委譲に変更
  4. すべての参照を更新

### Move Field（フィールドの移動）

- **目的**: フィールドをより適切なコンテキストに移動
- **適用シナリオ**: フィールドがそれを使用するメソッドと同じ場所にない場合
- **手順**:
  1. 移動先にフィールドを作成
  2. フィールドへの参照をすべて移動先への参照に変更
  3. 元のフィールドを削除

### Slide Statements（文のスライド）

- **目的**: 関連する文をまとめて配置
- **適用シナリオ**: 関連するコードが離れた場所にある場合、Extract Functionの準備
- **手順**:
  1. 移動したい文を特定
  2. 文が依存する要素の宣言より上には移動できないことを確認
  3. 参照される要素より下には移動できないことを確認
  4. 文を目的の場所に移動

### Split Loop（ループの分割）

- **目的**: 複数のことを行うループを1つずつ行う複数のループに分割
- **適用シナリオ**: ループが複数の処理を同時に行っている場合
- **手順**:
  1. ループをコピー
  2. 各ループで不要な部分を削除
  3. テストを実行
  4. パフォーマンスが重要な場合のみ後で結合を検討

### Replace Loop with Pipeline（ループをパイプラインに置き換え）

- **目的**: ループをfilter、map、reduceなどのコレクション操作に置き換え
- **適用シナリオ**: より宣言的で読みやすいコードにしたい場合
- **手順**:
  1. ループが作成するコレクションを特定
  2. filter、map、reduceなどのパイプライン操作で置き換え
  3. テストを実行

## Chapter 9: データの再編成

### Split Variable（変数の分割）

- **目的**: 複数回代入される変数を別々の変数に分割
- **適用シナリオ**: ループ変数やコレクタ以外で複数回代入される変数がある場合
- **手順**:
  1. 変数宣言と最初の代入時に新しい名前に変更
  2. 2回目の代入の前に新しい変数を宣言
  3. 各代入ごとにこれを繰り返す

### Rename Field（フィールド名の変更）

- **目的**: フィールド名をより適切なものに変更
- **適用シナリオ**: フィールドの意味がより明確になる名前がある場合
- **手順**:
  1. Encapsulate Recordを適用
  2. フィールド名を変更
  3. すべての参照を更新

## Chapter 10: 条件記述の簡略化

### Decompose Conditional（条件分岐の分解）

- **目的**: 複雑な条件分岐を名前付きメソッドに抽出
- **適用シナリオ**: 条件部分、then部分、else部分が複雑な場合
- **手順**:
  1. 条件部分をExtract Functionで抽出
  2. then部分をExtract Functionで抽出
  3. else部分をExtract Functionで抽出

### Consolidate Conditional Expression（条件式の統合）

- **目的**: 同じ結果を返す複数の条件を1つにまとめる
- **適用シナリオ**: 連続する条件チェックが同じアクションを実行する場合
- **手順**:
  1. 条件に副作用がないことを確認
  2. 条件をandまたはorで結合
  3. Extract Functionで条件を抽出して名前を付ける

### Replace Nested Conditional with Guard Clauses（ガード節による入れ子の条件記述の置き換え）

- **目的**: 特殊ケースの条件を早期リターンで処理
- **適用シナリオ**: 深い入れ子の条件があり、通常フローが不明確な場合
- **手順**:
  1. 特殊ケースをガード節として先頭に配置
  2. 各ガード節で早期リターンまたは例外を投げる
  3. すべてのガード節後にメインロジックを配置

### Replace Conditional with Polymorphism（ポリモーフィズムによる条件記述の置き換え）

- **目的**: 型やプロパティに基づく条件分岐をポリモーフィズムに置き換え
- **適用シナリオ**: 型に基づくswitch文や連続したif-else文がある場合
- **手順**:
  1. 条件の各分岐に対応するサブクラスを作成
  2. 各サブクラスに共通メソッドを作成
  3. 対応する分岐のコードを各メソッドに移動
  4. 条件分岐をメソッド呼び出しに置き換え

### Introduce Special Case（特殊ケースの導入）

- **目的**: null値や特殊ケースのチェックを専用クラスに置き換え
- **適用シナリオ**: null値や特殊値の同じチェックが複数箇所にある場合
- **手順**:
  1. 特殊ケースを表すクラスを作成
  2. 特殊ケースの振る舞いをクラスに実装
  3. 条件チェックを削除し、特殊ケースオブジェクトを使用

### Introduce Assertion（表明の導入）

- **目的**: コードの前提条件を明示的なアサーションで表現
- **適用シナリオ**: コードが正しく動作するために特定の条件が真である必要がある場合
- **手順**:
  1. 前提条件を特定
  2. アサーションを追加
  3. テストを実行してアサーションが適切であることを確認
