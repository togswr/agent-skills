# Determinism And Flaky Control

決定的テストを維持することは、テスト数を増やすより優先度が高い。

## 決定性の4軸

1. Time
- 現在時刻、タイムゾーン、サマータイム差分を注入で制御する。
- テストデータに絶対時刻を明示し、実行時刻依存を排除する。

2. Randomness
- 乱数シードを固定するか、乱数生成器を差し替える。
- 期待値は確率ではなく決定値で検証する。

3. External I/O
- ネットワーク、外部API、ファイルシステムの不確実性を隔離する。
- リトライやタイムアウト値をテスト専用に最適化しない。

4. Concurrency
- 非同期完了条件をイベントや状態で待つ。
- sleep 固定待機を排除し、明示的な同期条件を使う。

## フレーキー対応フロー

1. Capture
- 失敗ログ、入力、環境、実行順を記録する。

2. Quarantine
- 影響が大きい場合は一時隔離する。
- 隔離中はチケット、期限、担当を必須化する。

3. Reproduce
- 最小再現条件を作り、成功率ではなく再現条件を特定する。

4. Fix
- テストコードだけでなく、必要なら本番コードの設計改善も行う。

5. Prevent
- 同系統の失敗を検知するガード（lint, helper, CI policy）を追加する。

## 一時 skip のルール

- 許可条件:
  - ビジネス継続に対する被害が高く、即時恒久対策が不可能
- 必須条件:
  - 理由、期限、追跡ID、再有効化条件をコメントで残す
- 禁止条件:
  - 根因調査なしの恒常運用

## CI ガードレール

- フレーキー発生率を定期集計し、閾値超過でアラートする。
- 同一テストの連続隔離を検知したら優先度を引き上げる。
- 失敗の自動再試行は診断目的のみに限定する。

## 受け入れ基準

- 同じコミットでテスト結果が安定する。
- 失敗時に原因が仕様不一致か環境要因かを切り分けられる。
- 一時回避策より恒久対策の比率が高い。
